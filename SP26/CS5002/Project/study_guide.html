1. Overview: what you're actually building

You're given 20 patients. For each patient $i$ you know:

  • $x^{(i)}$ — total cholesterol level (mmol/L)
  • $y^{(i)}$ — diastolic blood pressure (mm Hg)

Your goal: predict blood pressure from cholesterol. Looking at the plot, the data is roughly a line, so try a linear model:

$$f(x) = ax + b$$

You need to find the best values of $a$ (slope) and $b$ (intercept). "Best" means the line passes as close as possible to all 20 points. To do that, you need:

  1. A measure of how bad a particular $(a,b)$ is → the cost function $g(a,b)$.
  2. An algorithm that finds the $(a,b)$ that minimizes the cost → gradient descent.
You're doing linear regression from scratch. Libraries like scikit-learn solve this in one call, but you're implementing the math yourself so you see what's under the hood.

2. The cost function $g(a,b)$

Step-by-step derivation (straight from the PDF)

For a given patient $i$ and a candidate line $(a, b)$, the prediction is $a x^{(i)} + b$, and the actual value is $y^{(i)}$. The error for that single patient is:

$$a x^{(i)} + b - y^{(i)}$$

Problem: errors can be positive or negative. If we just summed them, a +5 error on patient 1 and a −5 error on patient 2 would cancel out and look like "zero error". Bad.

Fix: square the error. Squares are always non-negative, so nothing cancels:

$$\left(a x^{(i)} + b - y^{(i)}\right)^2$$

Now add this up across all 20 patients to get one number describing how bad the line $(a,b)$ is overall:

$$g(a,b) = \sum_{i=1}^{n} \left(a x^{(i)} + b - y^{(i)}\right)^2$$

This is exactly equation (3) in the PDF. Minimizing $g$ → finding the best line.

Code for $g(a,b)$ (Q2)

def cost_function(a, b, x, y):
    n = len(x)
    total = 0.0
    for i in range(n):
        total += (a * x[i] + b - y[i]) ** 2
    return total

Straight translation of the formula: loop over patients, accumulate squared errors.

3. Gradient descent, intuition and mechanics

The hiking-in-fog picture

Imagine $g(a,b)$ as a landscape: $a$ is east–west, $b$ is north–south, altitude is the cost. You want to reach the lowest point. You can't see the whole landscape, but at any point you can feel the slope under your feet. Strategy: take a small step in the direction of steepest descent, then repeat.

Mathematically, the steepest-ascent direction at point $(a, b)$ is the gradient:

$$\nabla g(a,b) = \left(\frac{\partial g}{\partial a},\ \frac{\partial g}{\partial b}\right)$$

We want to go down, so we step in the opposite direction:

$$a_{\text{new}} = a - \alpha \frac{\partial g}{\partial a}, \qquad b_{\text{new}} = b - \alpha \frac{\partial g}{\partial b}$$

where $\alpha$ is the learning rate (step size). Repeat until the updates become tiny — that's your converged minimum.

How the code estimates the partials

You could compute $\partial g/\partial a$ analytically (it's $2\sum x^{(i)}(a x^{(i)} + b - y^{(i)})$), but the project uses a numerical approximation — the same definition of derivative you saw in calculus, just without taking the limit:

$$\frac{\partial g}{\partial a} \approx \frac{g(a + h,\ b) - g(a, b)}{h}$$

for some tiny $h$ (the code uses $h = 10^{-7}$). This is the forward difference.

def approx_partial_x(f, x, y, h=1e-7):
    return (f(x + h, y) - f(x, y)) / h

def approx_partial_y(f, x, y, h=1e-7):
    return (f(x, y + h) - f(x, y)) / h

The full loop

def gradient_descent_2d(f, x0, y0, alpha, epsilon, iter_max=100000):
    x, y = x0, y0
    iter = 0
    while iter < iter_max:
        grad_x = approx_partial_x(f, x, y)
        grad_y = approx_partial_y(f, x, y)
        x_new = x - alpha * grad_x
        y_new = y - alpha * grad_y
        # stop when both updates are tiny
        if abs(x_new - x) < epsilon and abs(y_new - y) < epsilon:
            x, y = x_new, y_new
            iter += 1
            break
        x, y = x_new, y_new
        iter += 1
    return x, y, iter
Why AND, not OR, in the stopping check?
It's possible for one parameter to be momentarily flat while the other is still moving. You only want to stop when both have settled. If you used or, you'd bail early and get a bad answer.

5. Why scaling matters (Q5 in depth)

The root cause: gradients on different magnitudes

Compute the partial derivatives analytically:

$$\frac{\partial g}{\partial a} = 2 \sum_{i=1}^{n} x^{(i)}\left(a x^{(i)} + b - y^{(i)}\right)$$ $$\frac{\partial g}{\partial b} = 2 \sum_{i=1}^{n} \left(a x^{(i)} + b - y^{(i)}\right)$$

Notice the difference: $\partial g/\partial a$ has an extra factor of $x^{(i)}$ inside the sum. In this dataset $x^{(i)}$ ranges from 100 to 300, so the gradient with respect to $a$ is roughly 100–300 times larger in magnitude than the gradient with respect to $b$.

What happens in the update

You apply the same learning rate $\alpha$ to both updates:

$$a \leftarrow a - \alpha \cdot \underbrace{\frac{\partial g}{\partial a}}_{\text{huge}}, \qquad b \leftarrow b - \alpha \cdot \underbrace{\frac{\partial g}{\partial b}}_{\text{small}}$$

You're stuck with an impossible tradeoff:

  • Pick $\alpha$ big enough for $b$ to move: the $a$ update is now gigantic, overshoots wildly, and the cost explodes (often to inf or NaN).
  • Pick $\alpha$ small enough to keep $a$ stable: $b$ barely moves, and you'd need millions of iterations.

Geometrically, the cost surface $g(a,b)$ is a steep narrow valley — very curved along $a$, very flat along $b$. Vanilla gradient descent with one $\alpha$ is bad at navigating such "ill-conditioned" surfaces.

The fix: standardize before fitting

Standardize (z-score) both variables:

$$x_s = \frac{x - \mu_x}{\sigma_x}, \qquad y_s = \frac{y - \mu_y}{\sigma_y}$$

Now both $x_s$ and $y_s$ have mean 0 and standard deviation 1. The gradients with respect to $a_s$ and $b_s$ end up on the same order of magnitude, the cost surface becomes roughly bowl-shaped (circular contours), and a single $\alpha$ works for both directions. Convergence in a few hundred iterations.

Short answer for the report
No — without scaling, gradient descent diverges because the partial derivative with respect to $a$ contains factors of $x^{(i)} \in [100, 300]$, making it dramatically larger than the partial with respect to $b$. A single learning rate can't serve both parameters. Z-scoring both variables equalizes the gradient magnitudes, making the cost surface well-conditioned and letting gradient descent converge from any reasonable starting point.

6. Converting scaled parameters back to original scale

Gradient descent on scaled data gives you $(a_s^*, b_s^*)$ satisfying $y_s \approx a_s^* x_s + b_s^*$. But you want the line for the original variables. Derive the conversion by substituting the z-score definitions:

$$\frac{y - \mu_y}{\sigma_y} = a_s\,\frac{x - \mu_x}{\sigma_x} + b_s$$

Solve for $y$:

$$y = \underbrace{a_s\,\frac{\sigma_y}{\sigma_x}}_{a^*}\, x + \underbrace{\mu_y + b_s \sigma_y - a_s\,\frac{\sigma_y \mu_x}{\sigma_x}}_{b^*}$$

Which is exactly what the code computes:

a_opt = a_s_opt * (y_std / x_std)
b_opt = b_s_opt * y_std + y_mean - a_s_opt * y_std * x_mean / x_std

The fit is noticeably worse. Visually, the line misses the shape of the data: the data points curve, whereas the line is straight. At low cholesterol (~100–175) the actual BP is nearly flat around 65 mm Hg; at high cholesterol (~250–300) BP shoots up toward 90+ mm Hg. Quantitatively the cost jumped from 28.86 (Section 1) to 179.23 — more than 6× worse.

Why different: the Section 1 data was genuinely linear, so a linear $f(x)$ matched the underlying relationship. The corrected blood-pressure monitor revealed that the true relationship is non-linear — roughly flat then curving upward. The algorithm is fine; the model class $f(x) = ax + b$ is too restrictive to represent a curved relationship. No line can fit curved data well.

What modifications can improve the model?

Replace the linear $f(x) = ax + b$ with something that can bend. Several reasonable options:

  • Quadratic: $f(x) = a x^2 + b x + c$ (what the code implements).
  • Higher-degree polynomial: cubic, quartic, etc. (More flexible but easy to overfit 20 points.)
  • Exponential: $f(x) = a\,e^{bx} + c$.
  • Piecewise linear: a low-slope line joined to a high-slope line.

The quadratic is a good first move: it adds just one extra parameter and can capture a single curve. See Section 8 for the implementation and results.

8. The quadratic fix

New cost function (three variables)

$$g(a, b, c) = \sum_{i=1}^{n} \left(a\,(x^{(i)})^2 + b\,x^{(i)} + c - y^{(i)}\right)^2$$

3D gradient descent

Same idea, one more partial derivative to track:

$$a \leftarrow a - \alpha\,\frac{\partial g}{\partial a}, \quad b \leftarrow b - \alpha\,\frac{\partial g}{\partial b}, \quad c \leftarrow c - \alpha\,\frac{\partial g}{\partial c}$$

Stop when all three updates are below $\epsilon$.

def gradient_descent_3d(f_3d, a0, b0, c0, alpha, epsilon, iter_max=100000):
    a, b, c = a0, b0, c0
    iter = 0
    while iter < iter_max:
        grad_a = approx_partial(f_3d, a, b, c, 0)
        grad_b = approx_partial(f_3d, a, b, c, 1)
        grad_c = approx_partial(f_3d, a, b, c, 2)
        a_new, b_new, c_new = a - alpha*grad_a, b - alpha*grad_b, c - alpha*grad_c
        if (abs(a_new-a) < epsilon and abs(b_new-b) < epsilon
                and abs(c_new-c) < epsilon):
            a, b, c = a_new, b_new, c_new
            break
        a, b, c = a_new, b_new, c_new
        iter += 1
    return a, b, c, iter

Scaling and the back-conversion

On scaled data we solve $y_s = a_s x_s^2 + b_s x_s + c_s$. Substituting $x_s = (x-\mu_x)/\sigma_x$ and $y_s = (y-\mu_y)/\sigma_y$, then expanding and collecting powers of $x$:

$$A^* = \frac{a_s\,\sigma_y}{\sigma_x^{\,2}}$$ $$B^* = \frac{b_s\,\sigma_y}{\sigma_x} - \frac{2\,a_s\,\sigma_y\,\mu_x}{\sigma_x^{\,2}}$$ $$C^* = \frac{a_s\,\sigma_y\,\mu_x^{\,2}}{\sigma_x^{\,2}} - \frac{b_s\,\sigma_y\,\mu_x}{\sigma_x} + c_s\,\sigma_y + \mu_y$$
